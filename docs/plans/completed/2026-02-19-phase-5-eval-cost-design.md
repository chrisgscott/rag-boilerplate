# Phase 5: Evaluation & Cost Tracking — Design Document

**Goal:** Built-in evaluation tooling (retrieval + answer quality metrics), per-query cost tracking, user feedback loop that feeds into evaluation.

**Architecture:** Server Actions for all mutations/runners. All data in Supabase (no external observability service). Model rates stored in DB with admin UI. Eval runner calls search directly for retrieval metrics, full pipeline for answer quality.

**Key Decisions:**
- Custom-built (not Langfuse) — keeps setup simple, gives buyers full control
- Multi-dimension answer quality rubric: faithfulness, relevance, completeness (1-5 each)
- Model rates in DB with admin page (not hardcoded)
- User feedback (thumbs up/down) converts to eval test cases
- Server Actions for eval runner (not background queue — YAGNI for typical 5-20 case test sets)

## 2. Cost Tracking

### Integration Point

The chat route's `onFinish` callback already has `usage` (inputTokens, outputTokens) and model ID. Add a `usage_logs` insert alongside the existing message save.

### Cost Calculation Flow

1. Chat completes → `onFinish` fires with token counts
2. Query `model_rates` table for current model's rates
3. Calculate: `embedding_cost = embedding_tokens × embedding_rate`, `llm_cost = (input × input_rate) + (output × output_rate)`
4. Insert into `usage_logs`

### Embedding Token Tracking

`hybridSearch()` calls the embedder internally. Modify it to return embedding token count from the OpenAI response so the route handler can include it in the usage log.

### Usage Dashboard (/usage)

- Summary cards: Total queries, Total cost, Avg cost/query, Projected monthly
- Time-series chart: daily cost over last 30 days
- Table: recent queries with per-query cost breakdown
- Server Components, filtered by org via RLS

### Model Rates Admin (/settings)

- Table of model rates with inline editing
- Pre-seeded with common model rates
- Add/remove models

## 3. Evaluation System

### Test Set Management (/eval — Tab 1: "Test Sets")

- CRUD for test sets (name, description)
- Within each test set: CRUD for test cases (question, expected answer, expected source document IDs)
- Document picker for expected source IDs (dropdown of org's documents)
- CSV import/export for bulk editing
- "Suggested from Feedback" section showing thumbs-down messages with "Convert to Test Case" button

### Eval Runner (/eval — Tab 2: "Run Evaluation")

- Select test set → click "Run"
- Two-phase execution per test case:

**Retrieval phase:**
- Call `hybridSearch()` directly with the question
- Compare retrieved chunk document IDs against expected source IDs
- Calculate Precision@k, Recall@k, MRR

**Answer quality phase:**
- Call `streamText()` with retrieved context + question to generate an answer
- Call LLM again as judge with rubric prompt:
  - Faithfulness (1-5): Is the answer grounded in the provided sources?
  - Relevance (1-5): Does the answer address the question?
  - Completeness (1-5): Does the answer cover all key points from the expected answer?
- Judge sees: question, expected answer, generated answer, retrieved sources

**Progress tracking:**
- `eval_results.status`: pending → running → complete
- UI polls for progress updates
- Config captured: model, top_k, similarity_threshold

### Results History (/eval — Tab 3: "Results History")

- Table of past runs with aggregate scores
- Pass/fail badges against targets (P@k ≥ 0.80, R@k ≥ 0.75, MRR ≥ 0.70)
- Expand run to see per-case breakdown
- Compare two runs side-by-side

## 4. User Feedback Loop

### Chat UI

- Thumbs up / thumbs down on each assistant message (appear on hover)
- Optional text comment on thumbs down ("What was wrong?")
- Saved to `message_feedback` via Server Action

### Feedback → Eval Pipeline

- /eval "Test Sets" tab shows "Suggested from Feedback" for unreviewed thumbs-down messages
- "Convert to Test Case" pre-fills question, lets dev add expected answer + sources
- `converted_to_test_case_id` tracks which suggestions have been handled

---

## 5. Testing Strategy

**Unit tests (Vitest):**
- Cost calculator: token counts + rates → correct costs
- Eval metrics: retrieved vs expected IDs → correct P@k, R@k, MRR
- Judge prompt builder: correct rubric prompt construction
- Model rates lookup: known models, unknown model fallback

**Integration coverage:** The eval runner itself is an integration test of the search pipeline.

**No Playwright e2e** for eval/usage pages — standard CRUD, not worth brittle test maintenance.
